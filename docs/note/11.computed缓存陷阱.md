# computed 在类属性中的缓存陷阱 【以下内容由AI生成，不确定是否正确】

## 问题描述

在类中通过属性赋值的方式使用 `computed`：

```ts
class DemoService {
  x = 1;
  computedX = computed(() => this.x);
}
```

当实例被 `reactive()` 包装后，`computedX` 在某些场景下无法响应 `x` 的变更。

## 根本原因

问题不在于 DI 框架，而是 Vue `computed` 内部的两条缓存检查路径：

### 路径一：无 effect 订阅时（能更新）

当 `computed` 没有被任何 `effect` 订阅时（比如纯 JS 中直接访问 `rt.computedX`），Vue 走 `globalVersion` 检查路径。每次访问 `.value` 时会检查全局版本号是否变化，如果变了就重新执行 getter，因此能拿到最新值。

```ts
const t = new DemoService();
const rt = reactive(t);

rt.computedX; // 1
rt.increaseX();
rt.computedX; // 2 ✅ 能更新
```

### 路径二：被 effect 订阅后（不能更新）

当组件模板渲染了 `computedX`（即 `{{ service.computedX }}`），`reactive` 代理自动解包 `ComputedRef` 的过程中，`computed` 被组件的渲染 `effect` 订阅。此时 Vue 切换到 `dep` 版本号检查路径。

由于 `computed(() => this.x)` 中的 `this.x` 不是响应式依赖（`this` 指向构造时的原始对象），`dep` 版本号永远不会递增。后续访问时 `computed` 认为自己是"干净的"，直接返回缓存值。

```ts
// 组件模板中渲染了 service.computedX
service.computedX; // 1
service.increaseX();
service.computedX; // 1 ❌ 不会更新
```

## 验证实验

通过 4 组对照实验精确证明了这一点：

| 场景 | 是否更新 | 原因 |
|------|---------|------|
| 模板渲染了 `computedX` | ❌ 否 | computed 被渲染 effect 订阅，走 dep 版本号路径 |
| 模板未渲染 `computedX` | ✅ 是 | computed 未被订阅，走 globalVersion 路径 |
| 不用 DI，直接 `new + reactive`，模板渲染 `computedX` | ❌ 否 | 同上，与 DI 无关 |
| 用 DI，模板不使用 service | ✅ 是 | computed 未被订阅，走 globalVersion 路径 |

## 影响范围

这个问题意味着：在类中通过 `computedX = computed(() => this.x)` 这种属性赋值方式使用 `computed`，一旦在组件模板中被渲染，就会失去响应式更新能力。

## 类型不一致问题

除了缓存陷阱之外，在类属性上直接使用 `computed` 还有一个类型层面的问题：

```ts
class DemoService {
  x = 1;
  computedX = computed(() => this.x); // 类型是 ComputedRef<number>
}
```

`computedX` 的 TypeScript 类型是 `ComputedRef<number>`，但当实例被 `reactive()` 包装后，`this.computedX` 实际上会被自动解包为 `number` 类型。这导致类型声明与运行时行为不一致：

- 编辑器中 `this.computedX` 提示类型为 `ComputedRef<number>`，IDE 会建议使用 `.value` 访问
- 但运行时 `this.computedX` 已经被 `reactive` 自动解包，实际是 `number`，直接使用 `.value` 反而会报错

这种类型不一致会在编辑器中误导开发者，增加心智负担。

## 结论：应使用 @Computed 装饰器

综合以上两个问题（缓存陷阱 + 类型不一致），在类属性上直接使用 `computed` 不是一个可靠的方案。应该使用 `@Computed` 装饰器来代替：

```ts
class DemoService {
  x = 1;

  // ❌ 不推荐：缓存陷阱 + 类型不一致
  computedX = computed(() => this.x);

  // ✅ 推荐：使用 @Computed 装饰器
  @Computed()
  get computedX() {
    return this.x;
  }
}
```

`@Computed` 装饰器的优势：

1. getter 的返回类型就是原始类型，类型声明与运行时行为一致
2. 在 getter 首次调用时才创建 `computed`，此时 `this` 已是 `reactive` 代理，能正确追踪响应式依赖
3. 通过 Symbol key 缓存 `computed` 实例，避免重复创建
